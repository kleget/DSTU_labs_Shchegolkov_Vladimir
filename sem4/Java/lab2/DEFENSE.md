# Лабораторная 2: Java Servlet API

Шпаргалка для защиты лабораторной. Основная идея работы: есть web-приложение на Java Servlet API, которое разворачивается в Tomcat под контекстом `/lab2`, обрабатывает HTTP-запросы из браузера и из JavaFX-клиентов.

## 1. Общая структура проекта

В лабораторной есть серверная часть и два JavaFX-клиента:

- `server/` - web-приложение с сервлетами, HTML-формами и `WEB-INF/users.xml`.
- `client_task3/AuthClientFx.java` - JavaFX-клиент для задания 3, отправляет GET/POST-запросы к `/auth`.
- `client_task5/CurrencyClientFx.java` - JavaFX-клиент для задания 5, получает данные от `/currency` и показывает таблицу + график.

Главные серверные классы:

- `server/src/main/java/lab2/AuthServlet.java` - авторизация и регистрация.
- `server/src/main/java/lab2/UserStore.java` - работа с XML-файлом пользователей.
- `server/src/main/java/lab2/SessionInfoServlet.java` - информация о текущей HTTP-сессии.
- `server/src/main/java/lab2/CurrencyServlet.java` - сервлет котировок валют и металлов.
- `server/src/main/java/lab2/CbrService.java` - запросы к XML-сервисам ЦБ РФ и разбор XML.

Маршруты сервлетов прописаны в:

```text
server/src/main/webapp/WEB-INF/web.xml
```

Там настроены:

- `/auth` -> `AuthServlet`
- `/session-info` -> `SessionInfoServlet`
- `/currency` -> `CurrencyServlet`
- стартовая страница -> `index.html`

Проект собирается как WAR-архив. Это задано в `server/pom.xml`:

```xml
<packaging>war</packaging>
```

Также там подключен `jakarta.servlet-api` со `scope=provided`, потому что реализацию Servlet API предоставляет Tomcat.

## 2. Как собрать и запустить

Самый простой запуск без ручного Tomcat:

```bash
mvn -f sem4/java/lab2/server/pom.xml jetty:run
```

Или из папки лабораторной:

```powershell
cd sem4/java/lab2
.\run.ps1
```

Для другого порта через скрипт:

```powershell
.\run.ps1 8081
```

После запуска открыть:

```text
http://localhost:8080/lab2/
```

Остановить сервер можно в терминале сочетанием `Ctrl+C`.

Важно: команда запуска не возвращает prompt, пока сервер работает. Это нормально. Нужно оставить терминал открытым и перейти в браузере по адресу выше.

Если в консоли Jetty пишет предупреждения вида `The XML schema ... could not be found`, это не ошибка запуска. Сервер все равно работает, если ниже есть строки `Started ... /lab2` или страница открывается в браузере.

Этот способ использует Maven Jetty plugin из `server/pom.xml`. Он поднимает локальный web-сервер, разворачивает приложение с контекстом `/lab2` и не требует вручную копировать WAR-файл в Tomcat.

Если порт `8080` занят, можно выбрать другой порт:

```bash
mvn -f sem4/java/lab2/server/pom.xml -Djetty.http.port=8081 jetty:run
```

Если сервер запущен не на `8080`, в JavaFX-клиентах нужно заменить порт в поле URL.

Если нужен именно классический запуск через Tomcat, можно собрать WAR и развернуть его вручную.

Сборка сервера:

```bash
cd sem4/java/lab2/server
mvn clean package
```

После сборки появляется WAR-файл:

```text
server/target/lab2-server-1.0.0.war
```

Его нужно развернуть в Tomcat. После развертывания приложение обычно доступно по адресу:

```text
http://localhost:8080/lab2/
```

Главные страницы для демонстрации:

```text
http://localhost:8080/lab2/index.html
http://localhost:8080/lab2/session.html
http://localhost:8080/lab2/currency.html
```

## 3. Задание 1: авторизация и регистрация

### Что требуется по заданию

Нужно создать простую систему авторизации/регистрации:

- HTML-страницу с формой логина и пароля.
- Форма отправляет GET-запрос на сервер.
- Сервлет проверяет логин и пароль.
- При успешной авторизации или регистрации выводится сообщение и текущее время сервера.
- При неправильном логине или пароле выводится номер попытки.
- После трех неудачных попыток авторизация в текущей сессии блокируется.
- Учетные данные пользователей хранятся на сервере в XML-файле.
- Пароль хранится не открытым текстом, а в виде хэша.

### Где это реализовано

HTML-форма:

```text
server/src/main/webapp/index.html
```

В форме:

```html
<form action="auth" method="get">
```

Поля формы:

- `action` - вход или регистрация.
- `login` - логин.
- `password` - пароль.

Сервлет:

```text
server/src/main/java/lab2/AuthServlet.java
```

Важные места:

- `doGet(...)` принимает GET-запрос.
- `process(...)` содержит общую логику авторизации/регистрации.
- `FAIL_KEY = "auth.fail"` - ключ, под которым в сессии хранится число неудачных попыток.
- `req.getSession(true)` получает или создает текущую HTTP-сессию.
- `store.register(login, password)` регистрирует пользователя.
- `store.verify(login, password)` проверяет логин и пароль.
- `LocalDateTime.now()` используется для вывода времени сервера.

Работа с пользователями:

```text
server/src/main/java/lab2/UserStore.java
```

XML-файл пользователей:

```text
server/src/main/webapp/WEB-INF/users.xml
```

Начальный пользователь:

```xml
<user login="admin" hash="8c6976e5b5410415bde908bd4dee15dfb167a9c873fc4bb8a81f6f2ab448a918"/>
```

Это логин `admin`, пароль `admin`, сохраненный как SHA-256 хэш.

### Что говорить преподавателю

Можно сказать так:

> В первом задании я реализовал форму авторизации и регистрации. Форма находится в `index.html` и отправляет GET-запрос на `/auth`. Запрос обрабатывает `AuthServlet`. Он получает параметры `action`, `login`, `password`, затем через класс `UserStore` проверяет пользователя в XML-файле `users.xml`. Пароли в XML не хранятся открыто, вместо них хранится SHA-256 хэш. Количество неправильных попыток хранится в `HttpSession`, поэтому блокировка работает именно для текущей сессии пользователя. После трех ошибок вход в этой сессии запрещается.

### Что показать

1. Открыть:

```text
http://localhost:8080/lab2/index.html
```

2. Выбрать регистрацию, ввести новый логин и пароль, отправить.
3. Показать сообщение об успешной регистрации и время сервера.
4. Вернуться назад, выбрать вход, ввести те же данные, показать успешную авторизацию.
5. Три раза ввести неверный пароль и показать блокировку текущей сессии.
6. Открыть `users.xml` и показать, что там хранится логин и хэш пароля.

## 5. Задание 3: GET/POST и JavaFX-клиент авторизации

### Что требуется по заданию

Нужно модифицировать сервлет из задания 1 так, чтобы он обрабатывал не только GET, но и POST-запросы. Также нужно создать JavaFX-приложение, которое:

- отправляет HTTP-запросы к сервлету авторизации;
- позволяет выбрать тип запроса: GET или POST;
- показывает ответ сервлета в своем окне.

### Где это реализовано

Сервлет:

```text
server/src/main/java/lab2/AuthServlet.java
```

В нем есть два метода:

```java
protected void doGet(HttpServletRequest req, HttpServletResponse resp)
protected void doPost(HttpServletRequest req, HttpServletResponse resp)
```

Оба вызывают:

```java
process(req, resp);
```

То есть логика авторизации не дублируется. GET и POST обрабатываются одинаково.

JavaFX-клиент:

```text
client_task3/AuthClientFx.java
```

Важные места:

- `ComboBox<String> method` - выбор `GET` или `POST`.
- `ComboBox<String> action` - выбор `login` или `register`.
- `sendRequest()` - формирует и отправляет запрос.
- Для POST задается заголовок:

```java
.header("Content-Type", "application/x-www-form-urlencoded")
```

- Для GET параметры добавляются к URL после `?`.
- Ответ сервлета выводится в `TextArea output`.

Можно сказать так:

JavaFX-клиент `AuthClientFx` позволяет выбрать метод GET или POST, ввести логин и пароль, отправить запрос на `/auth` и увидеть HTML-ответ сервлета в текстовом поле.

## 8. Коротко про ключевые технологии

### Что такое сервлет

Сервлет - это Java-класс на сервере, который принимает HTTP-запрос и формирует HTTP-ответ. В этой лабораторной сервлеты наследуются от `HttpServlet`.

### Что такое GET и POST

GET передает параметры в URL:

```text
/auth?action=login&login=admin&password=admin
```

POST передает параметры в теле запроса. В лабораторной POST-запрос отправляется с типом:

```text
application/x-www-form-urlencoded
```

### Что такое HttpSession

`HttpSession` позволяет хранить данные пользователя между запросами. HTTP сам по себе не хранит состояние, поэтому для счетчиков и блокировки используется сессия.

В лабораторной сессия используется:

- в задании 1 - для количества неудачных попыток входа;
- в задании 2 - для количества запросов к сервлету.

### Почему `users.xml` лежит в `WEB-INF`

Каталог `WEB-INF` закрыт для прямого доступа из браузера. Это значит, что пользователь не может просто открыть `users.xml` по URL. Но серверный Java-код может читать этот файл.

### Почему пароль хранится как хэш

Пароль не должен храниться открытым текстом. При регистрации пароль хэшируется через SHA-256, и в XML сохраняется хэш. При входе введенный пароль снова хэшируется, затем полученный хэш сравнивается с хэшем из XML.

### Что такое DOM

DOM - это способ работы с XML как с деревом узлов. В лабораторной DOM используется:

- в `UserStore.java` для чтения и записи `users.xml`;
- в `CbrService.java` для разбора XML-ответов Центрального банка.

## 9. Возможные вопросы преподавателя

### Почему в `AuthServlet` есть общий метод `process`?

Чтобы не дублировать код. GET и POST отличаются способом передачи параметров, но после получения параметров логика авторизации одинаковая.

### Где хранится количество неудачных попыток?

В текущей HTTP-сессии:

```java
session.setAttribute(FAIL_KEY, failCount);
```

Поэтому блокировка действует только в текущей сессии.

### Что произойдет после успешной авторизации?

Счетчик неудачных попыток сбрасывается:

```java
session.setAttribute(FAIL_KEY, 0);
```

### Почему `jakarta.servlet-api` имеет `scope=provided`?

Потому что Servlet API уже есть в Tomcat. При сборке он нужен для компиляции, но класть его внутрь WAR-файла не нужно.

### Почему для задания 5 используется CSV, а не HTML?

HTML удобен для браузера, но неудобен для разбора программой. CSV проще разобрать в JavaFX-клиенте: каждая строка содержит дату и значение через `;`.

### Почему график строится по номеру точки, а не по реальной дате?

В клиенте используется `NumberAxis`, поэтому по оси X ставится номер точки: 1, 2, 3 и так далее. Дата при этом отображается в таблице. Для лабораторной этого достаточно, потому что задача - показать динамику изменения значения.

### Что делать, если данные ЦБ не загрузились?

Нужно проверить:

- работает ли интернет;
- доступен ли сайт ЦБ РФ;
- правильно ли введен формат даты `dd.MM.yyyy`;
- запущен ли сервер Tomcat;
- правильно ли указан URL в JavaFX-клиенте.
